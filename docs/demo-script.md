# 60-second demo script

## 0:00–0:12 — Establish the boundary

“Benchling already validates entries, supports Notebook Check, and manages structured reviews. I did not build another version of that. I explored what happens after those checks repeatedly find the same problems.”

## 0:12–0:28 — Show the signal

Point to the project summary and recurring-friction list.

“These mock entry-check runs retain rule outcomes rather than scientific content. The most common finding—raw data not checked in—appears in half of the entries.”

## 0:28–0:43 — Show the evidence

Select `ATT-002` and point to affected runs.

“It spans several scientists and is associated with longer time-to-ready. That suggests workflow design friction, not an individual performance problem.”

## 0:43–0:53 — Show the action

Point to the suggested intervention.

“The system recommends adding a required raw-data attachment block to the assay template. Each recommendation stays traceable to a deterministic rule and the runs behind it.”

## 0:53–1:00 — Explain implementation realism

“This uses mock data. A real version could consume supported Benchling events and APIs, store low-sensitivity rule outcomes, and expose aggregates in an app or Insights. The open questions are permissions, retention, and whether the intervention measurably reduces rework.”
